Case Study- Chaikin Analytics- A DevOps Perspective

To negate the various challenges that we have discussed in our previous blog the process IT orgs should look up to is “DevOps” It offers solutions which focuses on the automation, re-usability, provisioning, commissioning and iterative improvements in the project development cycle.

We have prepared a case study on Chaikin-Analytics, one of the powerful stock market analysis tools.

We will be using this case study as an e.g to explain the challenges and problems faced by us during the development life cycle of this product and how the inculcation of DevOps tools and technologies have played a crucial role in tackling and resolving these challenges, making the application releases robust and reliable and achieving the intended results.

Following are the enhancements which were experienced after up-gradation to DevOps work culture:

Eliminated the difference in the development, testing and production environments in terms of machines/VMs configurations, OS version, external libraries versions etc using the Vagrant, Puppet.

  • Deploy new features in frequent iterations.
  • Faster build deployment time, Lowest cloud/application downtime Realistic Testing Environment.
  • Easily packaging the builds, fewer errors and issues.
  • No database Login required and the changes are updated automatically in the database using migrations.
  • Remove dependencies from all dev teams like development team, server team, database team, networking team.
  • Provision or migrate project from one cloud to another.

Eliminated the difference in development, testing and production environments in terms of machines/VMs configurations, OS versions, external library versions etc. using tools viz.vagrant and puppet:

The development and testing teams often have different operating systems, machine configurations when compared to production environments.

So to remove such differences, virtual development environments which are mirror images of the production environments with exact configurations, versions and settings etc can be replicated. Vagrant is an amazing tool which helps you achieve that.

Likewise, we created a centralized development environment using Vagrant for the Chaikin Analytics App and have used Puppet to provision and automate the builds and tasks based on the declarative and modular approach.

Using these tools, it takes only few minutes to create a new Chaikin Analytics environment in the existing cloud or in any other cloud space for any activity like development, testing etc.

Deployment of new features with frequent iterations:

We automated the deployment of builds using Puppet and Jenkins for Chaikin Analytics. We are following the continuous delivery approach so it has become extremely flexible for us to ship enhancements or releases to the production environment reliably, rapidly and judiciously.
Before using the DevOps tools and technology solutions, we had to wait for new deployments on production servers for months but automation has significantly reduced the time of deploying new features.

Leser build deployment time, Lowest cloud/application downtime:

Continuous Integration of code and packages with the central repository using the bitbucket, git, jenkins and sonatype nexus has helped us catalyse and automate the deployment process .

Prior to this automation process there was an inevitable downtime for the Chaikin Analytics Cloud which lasted about 2-4 hours but now the downtime has reduced considerably to just few mins.

Compatible Testing Environments:

Quality analysis is an essential ingredient for the successful performance of a product.

With the help of vagrant, we are able to provision compatible, generic and complementing ( to production) testing environments which negates the scenarios viz. “doesn’t work on my machine”

Whenever a new feature is introduced in the Chaikin Application, the testing environment is completely re-provisioned automatically in accordance with the targeted version of the application on which quality checks will be done.

This helped us to create stable deliverable packages and builds for final release.

Easily packaging the builds, fewer errors and issues:

As stated above that there are frequent possibilities of missing out on packages, components or other assets during the production roll out activities.

Continuous Integration of the project modules and code has helped in the continuous delivery of the project to QA phase and to check whether the business logics are adhered to or not.

In the Chaikin Analytics App we have used Git for merging development tasks with the master or other branches. Also we have used Jenkin to build all the project modules of required branch for an environment and these modules and packages are uploaded in the central repository i.e. Sonatype Nexus.

The complete packages and builds of required versions from this central repository are shipped automatically to the QA phase . Likewise, the packages and builds are deployed into the production environment from this central repository.

No database Login required and the changes are updated automatically in the database using migrations.

Another common problem is the migration of database updates from the development database to the production database.
If the changes are updated one by one manually to the production database, the probability of occurrence of errors increases many folds.Moreover this approach takes longer time resulting in longer downtime for the application.

Ruby Migrations is a suggested framework which helps to eliminate such problems. It is the most suitable framework to automate the deployment of database changes and updates using a single command. It helped us reduce the downtime considerably in Chaikin Application deployment activities.

Remove dependencies from all dev teams like development team, server team, database team, networking team

Having the dependencies on the resources or teams is one of the well known cause for the delay in the applications release.

Vagrant, Puppet, Chef, Jenkins, ruby migrations and git tools have played significant roles in automating every type of the module and tasks in the application.

So once individual tasks are automated then there will be no dependency on any resource or team during the build deployment in the production.
